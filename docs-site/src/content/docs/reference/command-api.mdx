---
title: Command API
description: Typed commands for relation lifecycle changes.
---

Commands describe lifecycle changes and carry audit metadata. The core crate
stays synchronous; async appears at persistence boundaries such as SQLx.

Use `CommandContext::new(actor)` for the common case, then add an idempotency
key or deterministic application metadata with the chainable builder helpers
when the command needs them.

| Command concern | Guidance |
| --- | --- |
| Actor | Use the application identity that caused the lifecycle change. |
| Idempotency key | Add one when a caller may retry the same command after an uncertain result. |
| Metadata | Add deterministic application context needed for audit, not request-local noise. |
| Timestamp | Pass the operation timestamp explicitly, or bind it once with `repo.at(now)`. |

Use `ApplyKeepsake::for_spec::<Spec>` when the relation is a typed
`RelationSpec`; use `ApplyKeepsake::new` when the relation id is dynamic. Pass
the command to `keepsake_sqlx::KeepsakeRepository::apply` so the lifecycle row
and SQL audit row commit together. Use `RevokeKeepsake::new` with
`KeepsakeRepository::revoke` for explicit revocation.

The SQLx adapter copies `CommandContext::metadata` into audit context
attributes. When `idempotency_key` is present, it is also stored as an
`idempotency_key` audit attribute unless metadata already supplied that key.
Repository duplicate prevention remains based on the active subject/relation
row, so use the command idempotency key as audit context rather than a separate
deduplication mechanism.
