---
title: Query Performance
description: Keep queries bounded and application-owned joins external.
---

Use relation ids for hot membership scans and subject kind/id for active
lookups. Join against application entity tables outside Keepsake so lifecycle
rows stay simple.

## Read Shapes

The SQLx adapter includes common read paths: active subject lookup, active
relation membership, due timed expiry, and fulfillment projection scans when the
`fulfillment-counters` feature is enabled.

Use `active_relations_for_subject_by_keys` when a request only needs a small
known set of relation keys for one subject. That keeps the query bounded without
making callers fetch all active subject relations and filter locally.

Indexes are part of that contract. Making them optional would reduce the
install-time diff while making production-scale query failures more likely. If
an application needs extra tenancy, retention, or partitioning indexes, add
those in the application migration layer.

## Pagination

Use keyset pagination for large membership scans. `active_membership_scan`
returns the first stable batch for a relation, and
`active_membership_scan_after` continues after a `MembershipCursor` built from
the last returned keepsake. Avoid offset pagination on hot relation membership
tables.

## Caching

Cache reads outside the SQL adapter when the workload needs it. Good cache
candidates are relation definitions by id or kind/name, active subject lookups
for request-scoped authorization checks, and membership scans with explicit
pagination inputs. Avoid caching open-ended scans without the full query shape
in the key.

The SQLx adapter includes an opt-in relation-definition cache behind the default
`cache` feature. This example assumes `pool` is a `sqlx::PgPool`.

```rust
use std::time::Duration;

use keepsake_sqlx::{KeepsakeRepository, LocalRelationCacheConfig};

let repo = KeepsakeRepository::new(pool)
    .with_local_relation_cache(LocalRelationCacheConfig::new(Duration::from_secs(60)));
```

The cache is local to the repository handle. Writes through that handle update
or invalidate cached relation definitions; writes from other processes may
remain cached until TTL expiry. Mutation paths such as `apply` still read and
lock relation definitions in Postgres, not from the cache.

Relation definitions are usually low-cardinality data. Start with a short TTL
and raise it only if relation lookups are hot enough to show up in traces or
query metrics.

In Kubernetes or another multi-replica deployment, the local cache is per pod.
That fits low-churn relation definitions with short TTLs.
If relation definition changes need cross-pod invalidation, implement the
`RelationCache` trait with an application cache such as Valkey, Redis, or a
control-plane-backed cache. Choose that topology in application code.

`KeepsakeRepository` is generic over its cache implementation and defaults to
`NoopRelationCache`. Applications can supply concrete, type-checked cache
adapters without trait-object dispatch.

Applications decide how to cache active subject lookups and membership scans
because they know partitioning, staleness tolerance, memory budgets, and
invalidation transport. Cache those only with the full relation id, subject,
cursor, limit, and any application partition scope in the key.

## Mutation Contention

Keep mutation paths authoritative and idempotent. Apply should return the
existing active keepsake for duplicate active commands, even if the relation has
since been disabled. New applies for disabled relations should fail. Relation
upsert should preserve the existing id for a natural key, and expiry workers
should tolerate already-terminal rows.

Large write volumes mainly contend on the relation definition row during applies
and expiry, because those operations take a shared lock to order themselves with
relation disable. This is intentional: disabling a relation is a policy change
that must not interleave unpredictably with new lifecycle transitions. If a
relation is extremely hot, keep disable operations rare and operationally
planned, and use multiple expiry workers with bounded batch sizes.
