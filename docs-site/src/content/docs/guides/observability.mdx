---
title: Observability
description: Record lifecycle outcomes without leaking application data.
---

Observability should show whether lifecycle work is healthy without turning
application subjects into metric labels.

Useful metric dimensions include command type, relation kind/name, transition
result, batch size, expired count, no-op category, and error category. These
values stay bounded and help answer whether applies, revokes, expiry workers,
and scans are behaving as expected.

Do not put opaque subject ids in metric labels. Use audit rows or trace spans
for subject id, request id, ticket id, and other high-cardinality values.

A practical dashboard starts with:

| Signal | Question it answers |
| --- | --- |
| apply/revoke result counts | are lifecycle commands succeeding |
| duplicate-prevented counts | are retries happening as expected |
| expiry batch size and duration | are workers keeping up |
| oldest due timed expiry age | are timed relations stuck active |
| repository error counts | is the adapter failing by category |

Keep business-level dashboards in the application. Keepsake emits lifecycle
outcomes; the product decides what those outcomes mean.
